Authentication Flow

OTP-based 2-step admin login and session management

ℹ️ Dashboard uses a 2-step OTP authentication. Admin enters phone number → receives OTP → verifies OTP → gets JWT token stored in localStorage.
StepEndpointMethodPayloadResponse
Step 1 — Request OTP /auth/signin POST { phone_number } { message: "OTP sent" }
Step 2 — Verify OTP /auth/verify-signin POST { phone_number, otp } { token, user: { role } }
Step 3 — All Requests Any /admin/* endpoint GET POST Authorization: Bearer <token> Protected response
🔐Server-Side

Backend Auth Middleware (5-Layer)

Every admin API route is protected by:

1. authenticate — Verifies JWT signature + expiry
2. JWT blacklist check — Logged-out tokens rejected
3. DB session validation — Session must exist in DB
4. requireRole(['admin']) — Role must be "admin"
5. Audit trail — Every admin action logged with adminId + timestamp
🪙Token

JWT Token Management

Token stored in localStorage. Auto-attached to all requests via Axios interceptor. On 401 response, user is redirected to login page automatically.

// Auto-attach in axios.js interceptor config.headers.Authorization = `Bearer ${localStorage.getItem('token')}`;

Security Architecture

How unauthorized access and abuse are prevented

✅ All destructive actions (block driver, suspend account, KYC approve/reject, issue refund) are protected by backend middleware. The dashboard cannot bypass any of these checks — they are enforced server-side.
🔒Backend Enforced

Driver Block — Can Anyone Do It?

No. Blocking a driver requires:

✅ Valid JWT token (not expired, not blacklisted)
✅ Active session record in database
✅ User role must be exactly "admin"
✅ All actions logged with adminId + timestamp
✅ 401/403 returned if any check fails
📋Backend Enforced

KYC Approval — Tamper-Proof

Backend KYC approval enforces additional validations beyond just auth:

✅ Document must exist and be in pending state
✅ Age verification — must be 18+
✅ PAN card format validation (regex)
✅ DL expiry check — expired DLs rejected
✅ Tamper detection on document images
✅ Per-route authorize('admin') middleware
🗂️Audit Trail

Admin Audit Log

Every admin action in the system is recorded in the database with:

📝 Admin user ID who performed the action
📝 Exact action type (verify_driver, reject_kyc, etc.)
📝 Target entity ID (driver/user/document)
📝 Timestamp (UTC)
📝 IP address
⚠️Known Limitation

Admin Role Levels

Currently all admin accounts have equal access — there is no super_admin tier differentiation enforced by the backend. Any admin can perform any admin action. This is a backend architecture limitation, not something the dashboard can fix unilaterally.

Backend Constraints & Known Limitations

Important facts about what the backend can and cannot do

ConstraintDetailDashboard Workaround
Analytics only accepts days param GET /admin/analytics/revenue only accepts ?days=N (1–365). Does NOT support startDate/endDate range filtering. Dashboard uses fixed period buttons: 7d, 30d, 90d, 365d
No GST/TDS breakdown endpoint Backend has no dedicated per-month GST breakdown API. Only runReport('financial') which sends an email report. TaxReports shows estimated GST based on revenue × 18%. Real monthly breakdown requires backend endpoint.
Reports are email-only POST /admin/reports/run/:type queues report generation and sends to admin email. No direct JSON response with report data. Buttons show toast "check admin email" after API call
No real-time websocket Backend uses REST only — no WebSocket/SSE for live updates. Dashboard data is snapshot-based. Manual Refresh buttons added to key pages (OperationalMetrics, Payouts)
Flat admin roles Backend only has role === 'admin' check — no sub-role differentiation (superadmin, finance-only, read-only). Dashboard cannot implement role-based UI hiding reliably — all admins see everything
